Premise\n\nThe case of choosing between a SPA and a SSR approach boils down to _who owns rendering the presentation_. Having the server render it, as it is the case in Turbo, decreases programming complexity a lot. So, taking the speed of light into account, the lowest possible latency is **22ms, one way**. N.B., that’s assuming a linear connection, no infrastructure (routing) costs involved. I’m going to round it up to **50ms** for our example here. Preparing the DOM\n\nTo implement our Optimistic UI solution, we first have to supply a \\`<template>\\` containing the \"successful\" HTML response already in our \\`<form>\\`:\n\n<https://stackblitz.com/edit/optimistic-ui-turbo-8-eb78vy?file=index.html%3AL109>\n\nIn a nutshell, we are constructing a Turbo Stream \"replace\" action containing the updated favorite button. Note the \\`{{optimisticSvg}}\\` placeholder here - in a Rails app this would be just rendered from a regular template, but here, the backend being NodeJS we have to jump through a few hoops to make that happen:\n\n<https://stackblitz.com/edit/optimistic-ui-turbo-8-eb78vy?file=index.js%3AL23>\n\nIn essence, we are \"mocking\" a server side template engine by reading the index.html as a string, and replacing the two \\`{{svg}}\\` and \\`{{optimisticSvg}}\\` placeholders with the respective SVG. Note that which one is the \"current\" and which the \"optimistic\" one depends on the server state, exemplified by the \\`favorite\\` variable.\n\n### Listening for Form Events\n\nTo trigger our optimistic UI update, we have to intercept the form submit. We can do this by listening for the \\`turbo:submit-start\\` event on our \\`<form>\\` element:\n\n<https://stackblitz.com/edit/optimistic-ui-turbo-8-eb78vy?file=app.js%3AL9>\n\nThe template of our Turbo Stream action already being present in the DOM, all we have to do is clone it and append it to the body, where it will be picked up by Turbo, and executed.\n\n### Reconciling the DOM Using Turbo Morph\n\nWhat though, if something goes wrong? Since Turbo 8, using morphs the answer is simple: Broadcast a \\`refresh\\` action, and the DOM will be reconciled to the old state. Note that you might want to provide some user feedback as well, in the form of a flash alert or similar.\n\nIn our case, we just need to send the respective Turbo Stream action from the backend, but in Rails you'll probably do this using [\\`broadcast\\_refreshes\\`](https://www.rubydoc.info/github/hotwired/turbo-rails/Turbo/Broadcastable):\n\n<https://stackblitz.com/edit/optimistic-ui-turbo-8-eb78vy?file=index.js%3AL56>\n\n**Note:** There is a small glitch we have to take care of, which is a [known bug in idiomorph](https://github.com/bigskysoftware/idiomorph/issues/15). In short, idiomorph doesn't morph the content of \\`<template>\\`s, so we have to take care of that ourselves:\n\n<https://stackblitz.com/edit/optimistic-ui-turbo-8-eb78vy?file=index.js%3AL37>\n\nThis snippet is **just** for swapping out the \"heart\" symbol with the opposite one - for preparing the Optimistic UI for the **reverse**
