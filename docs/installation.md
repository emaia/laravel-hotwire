## What the command does

In order, every `hotwire:install` invocation:

1. **Copies stubs** to `resources/` — `app.css`, `app.js`, the Stimulus bootstrap (`libs/stimulus.js`) and the controllers loader (`controllers/index.js`).
2. **Writes the loader stub** (`controllers/index.js`) generated from the catalog and the install flags. The first line carries an `// AUTO-GENERATED by hotwire:install — DO NOT EDIT MANUALLY.` marker so subsequent re-runs can safely regenerate it without prompting.
3. **Adds npm dependencies** to `package.json` (`devDependencies`). Always adds the three core packages (`@hotwired/stimulus`, `@hotwired/turbo`, `@emaia/stimulus-lazy-loader`); the catalog deps (echarts, leaflet, embla-carousel, tiptap, dropzone, maska, tippy.js, date-fns, sonner) follow the `--with-deps` / `--core-only` flags below.
4. **Injects the `@hotwire` Vite alias** into your `vite.config.{ts,mjs,js}` so user code can extend a vendor controller via `import CarouselController from '@hotwire/carousel_controller.js'`. Idempotent — re-running install never duplicates the key. If your config doesn't match the Laravel-stock `defineConfig({ ... })` shape, the snippet is printed for manual paste instead.
5. **Updates Laravel Idea metadata** — runs `hotwire:ide-json` to merge Hotwire Stimulus helper completions into the app's `ide.json`.
6. **Runs the package manager** (`bun/pnpm/yarn/npm install`, auto-detected from the lockfile). Interactive mode prompts; `--no-interaction` runs without prompting. `--skip-install` opts out entirely.
7. **Verifies view usage** — runs `hotwire:check` to detect any drift between the generated stub and the controllers actually referenced in views. In interactive mode the user is prompted to apply `--fix`; in `--no-interaction` mode the check reports findings and exits silently (use `--fix` for end-to-end automation).

Steps 3–7 are skipped when the user passes `--only=css` (CSS-only install).

## Dependency modes

| Command | Adds to `package.json` | Loader stub shape |
|---|---|---|
| `php artisan hotwire:install` | Core deps **plus every catalog dep** | Every package controller globbed — no exclusions |
| `php artisan hotwire:install --with-deps=carousel,chart,map` | Core deps **plus only the listed controllers' npm deps** | Zero-dep controllers + only the opted-in com-dep controllers; everything else excluded so `vite build` never resolves their imports |
| `php artisan hotwire:install --core-only` | Core deps **only** | Zero-dep controllers only; every com-dep controller excluded |

`--with-deps` accepts comma-separated values (`--with-deps=a,b,c`) and repeated flags (`--with-deps=a --with-deps=b`) — both forms work.

`--core-only` and `--with-deps` are mutually exclusive. Unknown controller names in `--with-deps` fail fast.

**Runtime cost is identical across the three modes.** Vite's dynamic-import code-splitting ships only the chunks for controllers that actually mount in the DOM. The trade-off is purely on the dev side — `node_modules` size, install time and `vite build` time scale with what's installed.

## Package manager step

`hotwire:install` runs your package manager by default.

| Flag | Effect |
|---|---|
| (no flag, interactive) | Prompts `Run bun install now?` (default yes) |
| (no flag, `--no-interaction`) | Runs the package manager automatically |
| `--skip-install` | Skips the package manager step — leaves dep fetching to the caller (useful in CI pipelines that wrap their own `npm ci` step) |

The detection order is `bun.lock` → `pnpm-lock.yaml` → `yarn.lock` → `package-lock.json`, falling back to `npm`. The same priority applies to `hotwire:check --fix`.

## The `@hotwire` Vite alias

`hotwire:install` injects this into your `vite.config.{ts,mjs,js}`:

```js
import { fileURLToPath } from 'node:url';

export default defineConfig({
    resolve: {
        alias: {
            '@hotwire': fileURLToPath(new URL('vendor/emaia/laravel-hotwire/resources/js/controllers', import.meta.url)),
        },
    },
    // ...your existing config...
});
```

The alias makes it easy to extend a vendor controller from your app code:

```js
// resources/js/controllers/gallery_controller.js
import CarouselController from '@hotwire/carousel_controller.js';

export default class extends CarouselController {
    // ...
}
```

The alias points directly at the `controllers/` folder so the common case (`extends CarouselController`) reads short. Internal shared helpers that live in the same folder (`_overlay.js`, `_focus_trap.js`, `_turbo_morph_recovery.js`) are reachable via the same alias (`@hotwire/_overlay.js`). Package-internal modules outside `controllers/` (e.g. `helpers/test_stimulus.js`, `libs/stimulus.js`) aren't exposed via `@hotwire`; they aren't documented public API.

The injection respects an existing `resolve:` block — your aliases are preserved, and `@hotwire` is appended into the same `alias: { ... }` literal. Custom config shapes that don't match `export default defineConfig({ ... })` are left untouched and the snippet is printed for manual paste.

See [extending-controllers.md](extending-controllers.md) for the three extension paths (alias, fork via `hotwire:controllers <name>`, vendor-relative import).

## The auto-generated loader stub

`resources/js/controllers/index.js` is **generated** from the catalog and your install flags — not copied bit-for-bit. The first line is fixed:

```js
// AUTO-GENERATED by hotwire:install — DO NOT EDIT MANUALLY.
// Re-run `php artisan hotwire:install` (or `hotwire:check --fix`) to regenerate.
```

`hotwire:install` recognises the marker and silently regenerates the file when you re-run with different flags. A hand-written `index.js` without the marker is treated as user-owned and never overwritten unless you pass `--force`.

The generated body is a single `registerControllers` call combining a glob of your app's controllers with a glob of the vendor's. Excluded com-dep controllers (per the install mode above) appear as `"!**/<name>_controller.js"` entries — Vite's `import.meta.glob` honours those exclusions, so excluded controllers never enter the build graph.

## Laravel Idea metadata

Laravel Hotwire ships package-level `ide.json` metadata for Blade components. Laravel Idea reads it from the Composer
package and provides completion/navigation for `<hw:*>` tags.

Stimulus helper completions are project-specific because local controllers can override package controllers. Generate the
app-level metadata with:

```bash
php artisan hotwire:ide-json
```

The command writes or merges `ide.json` in the app root. It preserves unrelated metadata, removes stale Hotwire
completion blocks, then adds package controllers plus local controllers discovered in `resources/js/controllers/**/*_controller.{js,ts}`.
Local controllers take precedence when they use the same identifier as a package controller.

`hotwire:install` runs `hotwire:ide-json` automatically for JS installs. Re-run it after adding, removing or renaming local
Stimulus controllers if you rely on PhpStorm/Laravel Idea completion for `stimulus_controller()`, `stimulus()->controller()`,
`stimulus_action()` or `stimulus_target()`.

## Detecting drift between install config and view usage

Suppose you install with `--with-deps=carousel` and later add `<hw:chart>` to a view. The build succeeds (chart is excluded from the stub), but Stimulus won't register the chart controller — the component renders inert.

To catch this:

```bash
php artisan hotwire:check
```

Reports:

```
Needs attention (2 issues):
  ✗  chart                excluded from loader stub  (used in views; re-run install with --with-deps including chart, or `hotwire:check --fix`)
  ✗  echarts ^6.1.0       missing from package.json (used by chart)
```

```bash
php artisan hotwire:check --fix
```

Applies all fixes — regenerates the stub to include `chart`, adds `echarts` to `package.json`, and runs your package manager. End-to-end resolution in one command.

`hotwire:install` runs `hotwire:check` automatically when `--core-only` or `--with-deps` is set (the default mode has no exclusions, so there's no drift to detect). In interactive mode the user is prompted to apply `--fix`; in `--no-interaction` mode the check reports but does not act unless `--fix` is also passed.

## CI / automation recipe

End-to-end install with zero prompts:

```bash
php artisan hotwire:install --with-deps=modal,dropdown --fix --no-interaction
```

This: scaffolds, adds core + selected deps + `@hotwire` alias, generates the trimmed loader stub, runs `bun install`, runs `hotwire:check --fix` (regenerates the stub and adds any drifted npm deps based on existing view usage), runs `bun install` again if check added new deps, and never prompts.

For the "I'll handle the package manager myself" CI pattern:

```bash
php artisan hotwire:install --with-deps=modal,dropdown --fix --skip-install --no-interaction
bun ci
```

`--skip-install` is propagated to the post-install `hotwire:check` call as well, so neither phase invokes the package manager.

## Flag reference

| Flag | Purpose |
|---|---|
| `--force` | Overwrite existing scaffolded files without prompting. Hand-written files (no `// @hotwire-package` marker for controllers; no `// AUTO-GENERATED` marker for the loader stub) are still protected unless `--force` is passed. |
| `--only=js` / `--only=css` | Install only the JS or CSS scaffold. Skips npm deps, Vite alias and post-install check when CSS-only. |
| `--with-deps=<list>` | Add npm deps only for the listed controllers. Comma-separated or repeatable. Validates each name against the catalog. |
| `--core-only` | Add only core deps (stimulus, turbo, dynamic-loader). Excludes every com-dep controller from the loader stub. Mutually exclusive with `--with-deps`. |
| `--skip-install` | Do not run the package manager after writing `package.json`. Also propagated to the post-install `hotwire:check`. |
| `--fix` | Auto-apply `hotwire:check --fix` during the post-install verification. Pairs with `--no-interaction` for end-to-end automation. |
| `--no-interaction` | Standard Laravel flag — suppresses all prompts. Combined with `--fix`, applies all fixes silently; without `--fix`, reports findings but does not act. |

## What changes in your app after install

Files written (or modified) under your project root:

- `resources/js/app.js`
- `resources/js/libs/stimulus.js`
- `resources/js/controllers/index.js` ← auto-generated, has the marker
- `resources/css/app.css`
- `package.json` ← `devDependencies` updated
- `vite.config.{ts,mjs,js}` ← `@hotwire` alias added (if matchable)
- `ide.json` ← Laravel Idea Stimulus helper completions merged

None of your existing code is touched outside of those entries — the alias injection respects your existing `resolve:` block, npm deps are merged into `devDependencies` without replacing unrelated keys, and the loader stub is recognized as generated so re-running install does not require `--force`.

## Troubleshooting

**Build fails with `Cannot resolve "echarts"` (or another catalog dep).** You installed with `--core-only` or `--with-deps=<list>` that did not include the controller using that dep, but then a view started using `<hw:chart>` (or another com-dep component). Run `php artisan hotwire:check --fix` to add the missing dep and regenerate the loader stub.

**Build resolves a vendor controller's deps even though I didn't opt in.** This means the loader stub is missing the `// AUTO-GENERATED` marker — either it was hand-edited or pre-dates `0.32.0`. Run `php artisan hotwire:install --force` to regenerate it with the appropriate exclusions for your current flags.

**The Vite alias snippet was printed instead of injected.** Your `vite.config.js` doesn't match `export default defineConfig({ ... })` (custom shape, multiple exports, etc.). Paste the printed snippet manually inside `defineConfig({ ... })` — `hotwire:install` is intentionally conservative and does not attempt to merge into non-stock configs to avoid corrupting them.

**Re-running `hotwire:install` does not regenerate `index.js`.** Either the file already matches the generated output exactly (no-op), or the marker comment has been removed. Add the marker line at the top of the file (or pass `--force`) to re-enable automatic regeneration on flag changes.

**Re-installing under path-repo (composer local symlink) fails to resolve a vendor controller's bare imports.** Vite resolves bare imports from the real path of the file, which lives in the package source. Either install the missing dep in the package's own `node_modules` (when you're testing locally), or add `resolve: { preserveSymlinks: true }` to your Vite config so the alias path is honoured during resolution.
